## Vision

### 10x Check

The 10x version of Keplore is not a co-worker tool — it is institutional memory
infrastructure for industrial companies. Every skill built through delivery work
becomes a durable organizational asset. It compounds across time (skills survive
expert departures), geography (skills propagate from one factory site to another),
and eventually across companies (a shared commons of non-proprietary industrial
knowledge that any customer starts from).

Revenue model at 10x: skills library subscription as an HR/operations asset, not
just an AI tool. New employee onboarding 3x faster. New buyer personas (HR, ops)
alongside engineering.

### Platonic Ideal

A Geely engineer opens Keplore on their first day. They describe a problem in
natural language: "We're adapting last year's motor inspection model for the new
battery cell line — different camera geometry, different defect types."

The system responds: "I found your motor inspection skill from March. The key
adaptations are X, Y, Z. Here are three approaches. Approach A reuses 80% of
the existing model — fastest path, 2 days. Approach B rebuilds the feature
extraction layer — better accuracy, 1 week. Which fits your timeline?"

The engineer didn't write a prompt. Didn't configure anything. The system already
knew the company's history and offered a concrete path forward with full visibility
into which skills it was using and where it was uncertain. That feeling — a very
smart colleague who already knows your domain and your history — is the product.

## Architecture Shift (from /office-hours design → CEO review correction)

The original design proposed YAML config files + custom embedding-based routing.
Updated to Claude Agent SDK native skills paradigm:
- Skills are SKILL.md (interface) + scripts/ (executable code) + data/ (static resources)
- Claude autonomously discovers and invokes skills via description matching — no custom router
- Expert roles = skill directories grouped by domain
- Version control via git (SKILL.md is just a file)
- Skill health monitoring enabled by bundled `tests/validate.py` scripts

This is architecturally cleaner and builds on Anthropic's trained-in behavior,
not a custom retrieval system.

## HARD BLOCKER: Resolve Before v1 Build Starts

**Confirm Geely deployment model (cloud vs. on-prem).** This is not an open question
to answer during implementation — it changes the core architecture. If on-prem is
required: SI Onboarding defers to v1.5, and the on-prem fallback router adds ~2 weeks.
Do not start v1 engineering until this is answered.

**Validate Claude Agent SDK skill selection signal.** Skill Transparency requires the
SDK to expose a usable skill-selection signal (which skills Claude drew on, at what
confidence). Confirm this is available in the SDK before committing to the transparency
UX. If the signal is unavailable or opaque, the fallback is to surface skill names
without confidence levels (degraded but still useful).

## Key Open Decisions

1. **Geely deployment model: on-prem vs. cloud?** This gates the entire v1 architecture.
   - If **cloud**: use Claude API + Claude Agent SDK natively. Skill selection is handled
     by Claude's trained behavior. Straightforward.
   - If **on-prem / data sovereignty**: Claude API is not usable. Fallback architecture:
     deploy Qwen2.5 (or similar) via vLLM on customer infrastructure. Skill discovery
     requires a lightweight router (embedding similarity on SKILL.md descriptions using
     a locally-run embedding model, e.g., `sentence-transformers/all-MiniLM-L6-v2`).
     The SKILL.md format is model-agnostic — content doesn't change, only the invocation
     layer differs. This adds ~2 weeks to v1 and requires containerized deployment packaging.
   Action: Confirm with Geely before any v1 implementation starts.

2. Aitomatic differentiation: what's the answer before Series A?
   Current thesis: skills-as-complete-packages (not just knowledge retrieval),

## Architecture: Expert Role Structure

```

    halcon-scripting/
    plc-integration/

        SKILL.md

      arm-calibration/
        SKILL.md
        scripts/calibrate.py
  china-manufacturer-tenant/
    vision-model-engineer/

        SKILL.md
        scripts/retrain.py
        data/training_config.yaml
  idaho-si-tenant/
```

Claude loads all SKILL.md frontmatter at session start. For a Geely session,
Claude sees the `geely-tenant/` skills + `_shared/` skills. It autonomously
selects which to invoke based on the task description matching the `description:` field.

## Skill Transparency — Implementation Spec

Every output includes a structured footer:

```
─── Skills invoked ────────────────────────────────────
✓ magnetic-defect-threshold-tuning (high confidence)
✓ camera-calibration               (medium confidence)
⚠ thermal-compensation             (no skill — used general AI)
───────────────────────────────────────────────────────
```

Confidence is derived from the Claude Agent SDK's skill selection signal:
- High: Claude explicitly selected this skill and it was the primary input
- Medium: skill was selected but Claude also drew on general knowledge
- Gap (⚠): Claude flagged that relevant domain knowledge was missing and fell back

In the on-prem fallback (local model + embedding router):
- High: cosine similarity ≥ 0.75
- Medium: similarity 0.5–0.75
- Gap: similarity < 0.5 or no skill in top-3 reaches 0.5

## SI Onboarding — Spec

For system integrators (Idaho and future US SIs):

**Input accepted at signup:**
- Past project code repositories (GitHub/GitLab URL or zip)
- Technical documentation (PDF, Word, markdown)
- Prior conversation/chat history with AI tools (exported JSON)
- Manual description of past project types (free text)

**Process (effort: M, not S):**
1. Keplore engineer runs an extraction session: feeds materials to Claude, which
   drafts SKILL.md files for each identifiable skill pattern
2. SI engineer reviews drafted skills, corrects domain details, approves
3. Skills are added to the SI's tenant library
4. Target: 10-30 starter skills in 2-4 hours of collaborative onboarding session

**Quality gate:** SI engineer must explicitly approve each SKILL.md before it goes
live. No auto-publish of extracted skills.

## Shared Skills Library — Gating Condition

Build the commons layer when: (a) a second customer independently requests a skill
that already exists for another customer in non-proprietary form, OR (b) Keplore's
engineers have built the same generic skill pattern 3+ times across different tenants.
Do not build speculatively. The `shared: true` frontmatter flag is already in the
design — the infrastructure cost is near-zero when the trigger condition is met.
